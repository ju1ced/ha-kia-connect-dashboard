# Runtime Mapping Health

## Purpose

Runtime mapping health defines how the dashboard should detect and explain
missing, unknown, or unavailable mapped entities after installation in Home
Assistant.

The dashboard already validates the static mapping contract in CI. Runtime health
covers what happens inside a user's Home Assistant instance.

## Health States

Use these states when designing future health cards or helper entities:

- `ok`: the mapped entity exists and has a usable state.
- `missing`: the mapped entity ID does not exist in Home Assistant.
- `unknown`: the mapped entity exists but reports `unknown`.
- `unavailable`: the mapped entity exists but reports `unavailable`.
- `not_configured`: the mapping key exists but intentionally points to a
  placeholder or disabled entity.
- `not_checked`: the mapping key is valid, but no runtime state export was
  provided to check it against.

## Check Script

Use `scripts/check_runtime_mapping_health.py` to compare
`dashboard/templates/entities.yaml` with a Home Assistant state export.

Run a static summary without Home Assistant state data:

```bash
python scripts/check_runtime_mapping_health.py
```

Run a runtime check with exported state data:

```bash
python scripts/check_runtime_mapping_health.py --states-json ha-states.json
```

Use strict mode when you want the command to fail for missing, unknown, or
unavailable entities:

```bash
python scripts/check_runtime_mapping_health.py --states-json ha-states.json --strict
```

## Home Assistant State Export

The script accepts either of these JSON shapes:

```json
[
  {
    "entity_id": "sensor.kia_ev6_battery_level",
    "state": "80"
  }
]
```

```json
{
  "sensor.kia_ev6_battery_level": "80"
}
```

A full export can come from the Home Assistant REST `/api/states` endpoint or a
small hand-built export containing only Kia-related entities for the first test.

## Example Contract

`dashboard/templates/mapping_health.example.yaml` documents the expected runtime
health shape. It is not included by the dashboard yet and should not be treated
as live Home Assistant configuration.

The example file defines:

- Supported health states and severity levels.
- Required fields for future health rows.
- Example rows for mapped, unavailable, and optional entities.

## First Surface

Settings owns the first runtime health surface. It should summarize mapping
health before inline alerts are added to Battery, Vehicle, Climate, Energy,
Location, or Overview.

The current Settings maintenance card reads the mapped
`settings.mapping_health_summary` entity. The check script is the repository-side
source of truth until that summary is backed by a Home Assistant helper or
template sensor.

## Inline Alerts

`docs/inline-unavailable-alerts.md` defines where missing, unknown, or
unavailable entities may appear outside Settings. Keep Settings as the complete
diagnostics view, and use inline alerts only for critical page-blocking values.

## Future Data Shape

A future runtime health helper should expose at least:

- Logical key, such as `battery.level`.
- Mapped entity ID, such as `sensor.kia_ev6_battery_level`.
- Health state.
- Last checked timestamp.
- Short remediation hint.

## Rendering Rules

- Show summary health in Settings first.
- Keep detail pages focused on vehicle data unless a mapped entity blocks the
  page from being useful.
- Prefer clear text such as `Check dashboard/templates/entities.yaml` over raw
  stack traces or integration internals.
- Do not enable service actions based only on a display entity existing.

## Open Questions
